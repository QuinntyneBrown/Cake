# L2 Requirements — Cake CLI

## Traceability

Each L2 requirement traces to an L1 requirement and includes acceptance criteria.

---

## L1-REQ-001: CLI Framework and Host Infrastructure

### L2-REQ-001.1: System.CommandLine Root Command

The CLI shall define a root command using `System.CommandLine` that serves as the entry point for all subcommands.

**Acceptance Criteria:**

- A root command is registered with a description (e.g., `"Cake CLI — build scaffolding and code generation"`).
- Running the CLI with `--help` displays available subcommands and global options.
- Running the CLI with `--version` displays the current CLI version.

---

### L2-REQ-001.2: Dependency Injection via Microsoft.Extensions.DependencyInjection

The CLI shall configure a DI container using `Microsoft.Extensions.DependencyInjection` and register all services, command handlers, and integrations.

**Acceptance Criteria:**

- A `IServiceCollection` is configured at startup with all required services.
- Command handlers receive dependencies via constructor injection.
- Service lifetimes (Singleton, Scoped, Transient) are appropriate for each registration.

---

### L2-REQ-001.3: Logging via Microsoft.Extensions.Logging

The CLI shall configure structured logging using `Microsoft.Extensions.Logging`.

**Acceptance Criteria:**

- Console logging is enabled by default.
- A global `--verbosity` option controls log level (`Quiet`, `Minimal`, `Normal`, `Verbose`, `Diagnostic`).
- Command execution, errors, and key operations produce log output at appropriate levels.

---

### L2-REQ-001.4: Configuration via Microsoft.Extensions.Configuration

The CLI shall load configuration from standard sources using `Microsoft.Extensions.Configuration`.

**Acceptance Criteria:**

- Configuration is loaded from `appsettings.json` (if present) and environment variables.
- Configuration values are injectable via `IOptions<T>` or `IConfiguration`.
- Missing configuration files do not cause startup failure.

---

## L1-REQ-002: Cake Build Integration

### L2-REQ-002.1: Cake Frosting Dependency

The scaffolded build project shall reference Cake Frosting NuGet packages to enable Cake-based build orchestration.

**Acceptance Criteria:**

- The generated `build.csproj` includes a `PackageReference` to `Cake.Frosting`.
- The generated `Program.cs` contains a minimal Cake Frosting host setup (e.g., `CakeHost` builder pattern).
- The scaffolded build project compiles without errors using `dotnet build`.

---

### L2-REQ-002.2: CLI Extends Cake Capabilities

The CLI itself shall reference Cake packages and provide Cake-aware utilities or aliases that extend Cake's built-in functionality.

**Acceptance Criteria:**

- The CLI project references `Cake.Core` and `Cake.Common`.
- At least one custom Cake alias or utility is exposed through the CLI's integration layer.
- The CLI can invoke Cake tasks programmatically.

---

## L1-REQ-003: Code Generator and Git Integration

### L2-REQ-003.1: QuinntyneBrown.Git.Core Integration

The CLI shall register and use `QuinntyneBrown.Git.Core` for git operations.

**Acceptance Criteria:**

- `QuinntyneBrown.Git.Core` services are registered in the DI container.
- Git operations (e.g., init, status) are available to command handlers via injected services.

---

### L2-REQ-003.2: QuinntyneBrown.CodeGenerator.DotNet Integration

The CLI shall register and use `QuinntyneBrown.CodeGenerator.DotNet` for .NET code generation.

**Acceptance Criteria:**

- `QuinntyneBrown.CodeGenerator.DotNet` services are registered in the DI container.
- .NET artifacts (solutions, projects, source files) can be generated via the code generator.

---

### L2-REQ-003.3: QuinntyneBrown.CodeGenerator.Angular Integration

The CLI shall register and use `QuinntyneBrown.CodeGenerator.Angular` for Angular code generation.

**Acceptance Criteria:**

- `QuinntyneBrown.CodeGenerator.Angular` services are registered in the DI container.
- Angular artifacts (components, modules, services) can be generated via the code generator.

## L1-REQ-004: Claude Skill Installation Command

### L2-REQ-004.1: `install-skill` Command Definition

The CLI shall expose an `install-skill` command (e.g., `cake-cli install-skill`) that installs a Claude skill into the current working directory.

**Acceptance Criteria:**

- The command is registered as a subcommand of the root command with the name `install-skill`.
- Running `cake-cli install-skill --help` displays usage and description.
- The command can be invoked with no required arguments (operates on the current directory).

---

### L2-REQ-004.2: Claude Skill File Generation

When `install-skill` is executed, the CLI shall create the Claude skill file(s) in the appropriate location within the current working directory.

**Acceptance Criteria:**

- A `.claude/skills` directory (or equivalent Claude skill location) is created if it does not exist.
- A skill file is written that describes how to use the Cake CLI (commands, options, usage patterns).
- If the skill file already exists, the user is warned and the file is overwritten or the command fails gracefully (with a `--force` option to overwrite).
- The installed skill file is valid and usable by Claude Code.

---

### L2-REQ-004.3: Skill Content

The installed Claude skill shall contain instructions that teach Claude Code how to invoke and use the Cake CLI.

**Acceptance Criteria:**

- The skill describes all available CLI commands and their options.
- The skill includes example invocations.
- The skill references the CLI's binary/command name correctly.

---

## L1-REQ-005: Project Scaffolding Command

### L2-REQ-005.1: `create` Command Definition

The CLI shall expose a `create` command (e.g., `cake-cli create`) with a required `--name` option that specifies the project folder name.

**Acceptance Criteria:**

- The command is registered as a subcommand of the root command with the name `create`.
- A `--name` option (or positional argument) is required and specifies the folder name.
- Running `cake-cli create --help` displays usage, options, and description.
- If `--name` is omitted, the CLI displays an error message indicating the option is required.

---

### L2-REQ-005.2: Root Folder Creation

When `create` is executed, the CLI shall create a folder with the name specified by the `--name` option in the current working directory.

**Acceptance Criteria:**

- A directory named `{name}` is created in the current working directory.
- If the directory already exists, the command fails with a descriptive error (unless `--force` is specified).
- The folder name supports valid filesystem characters.

---

### L2-REQ-005.3: Build Folder and Solution Scaffolding

Inside the created root folder, the CLI shall create a `build` subfolder containing a solution file (`build.sln`), a project file (`build.csproj`), and a program file (`Program.cs`).

**Acceptance Criteria:**

- A `build` directory is created at `{name}/build/`.
- A solution file `build.sln` is created at `{name}/build/build.sln`.
- A project file `build.csproj` is created at `{name}/build/build.csproj`.
- A program file `Program.cs` is created at `{name}/build/Program.cs`.
- The `build.sln` includes the `build.csproj` as a project reference.
- The `build.csproj` targets a supported .NET version (e.g., `net8.0` or later).
- The `build.csproj` references `Cake.Frosting` as a NuGet package dependency.
- `Program.cs` contains a valid Cake Frosting entry point that compiles and runs.
- Running `dotnet build` in the `build` folder succeeds without errors.

---

### L2-REQ-005.4: Claude Skill Auto-Installation in Scaffolded Project

After creating the build folder structure, the `create` command shall automatically install the Claude skill (as defined in L2-REQ-004.2) into the `{name}/build/` directory.

**Acceptance Criteria:**

- The Claude skill is installed in the `{name}/build/` directory as part of the `create` command execution.
- The skill installation follows the same behavior as the standalone `install-skill` command.
- The skill files are present and valid after the `create` command completes.
- No user interaction is required for the skill installation during `create`.

---

## L1-REQ-006: Add Project Command

### L2-REQ-006.1: `add-project` Command Definition

The CLI shall expose an `add-project` command (e.g., `cake-cli add-project`) with a required `--git-url` option specifying a git repository URL and a required `--projects` option accepting one or more .NET project names that exist within that repository.

**Acceptance Criteria:**

- The command is registered as a subcommand of the root command with the name `add-project`.
- A `--git-url` option is required and accepts a git repository URL string.
- A `--projects` option is required and accepts one or more .NET project names (e.g., `--projects MyLib.Core MyLib.Api`).
- Running `cake-cli add-project --help` displays usage, options, and description.
- If `--git-url` or `--projects` is omitted, the CLI displays an error indicating the missing required option.

---

### L2-REQ-006.2: Build Directory Validation

When `add-project` is executed, the CLI shall validate that the current working directory is a valid cake build directory containing a `build.csproj` with a `Cake.Frosting` package reference and a `Program.cs` file.

**Acceptance Criteria:**

- The command checks for the existence of a `.csproj` file in the current directory.
- The command checks for the existence of a `Program.cs` file in the current directory.
- If validation fails, the command exits with a non-zero exit code and a descriptive error message.
- If validation succeeds, the command proceeds to modify the build program.

---

### L2-REQ-006.3: Program.cs Modification with Solution Generation Logic

When `add-project` is executed in a valid build directory, the CLI shall regenerate `Program.cs` to include a Cake Frosting task (`GenerateSolutionTask`) that, when the build program runs, creates a solution file in the root directory (parent of the build directory) if no `.sln` or `.slnx` file exists, clones the specified git repository, and adds the specified .NET projects to the solution.

**Acceptance Criteria:**

- `Program.cs` is rewritten with the following generated code structure:
  - A `BuildContext` class extending `FrostingContext` with a `RootDirectory` property pointing to the parent of the build directory.
  - A `GenerateSolutionTask` Cake Frosting task class that:
    - Checks if any `.sln` or `.slnx` file exists in the root directory; if so, skips generation.
    - Clones the git repository (from `--git-url`) into `{rootDir}/src/{repoName}/` if not already present.
    - Creates a new solution file in the root directory using `dotnet new sln`.
    - Searches for each specified project's `.csproj` file recursively within the cloned repository.
    - Adds each found `.csproj` to the solution using `dotnet sln add`.
  - A `DefaultTask` with `[IsDependentOn(typeof(GenerateSolutionTask))]`.
- The repository name is derived from the git URL (last segment without `.git` extension).
- The solution name is derived from the root directory name at runtime.
- The generated `Program.cs` compiles without errors when `dotnet build` is run in the build directory.

---

### L2-REQ-006.4: Template Provider for Generated Program.cs

The CLI shall provide a template method in `BuildProjectTemplateProvider` (or a new template provider) that generates the `Program.cs` content parameterized by git URL, repository name, and project names.

**Acceptance Criteria:**

- A template method accepts a git URL string and an array of project name strings.
- The template method returns a complete, compilable `Program.cs` string containing the `GenerateSolutionTask` logic.
- The repository name is extracted from the git URL (last path segment minus `.git`).
- The generated code includes all necessary `using` directives (`System.Diagnostics`, `Cake.Core`, `Cake.Frosting`, etc.).
- The template output is deterministic for the same inputs.

---

### L2-REQ-006.5: Build Program Modifier Service

The CLI shall provide a service (`IBuildProgramModifier` / `BuildProgramModifier`) that reads an existing `Program.cs`, generates the updated content using the template provider, and writes it back to disk.

**Acceptance Criteria:**

- The service is registered in the DI container.
- The service accepts a build directory path, a git URL, and an array of project names.
- The service reads the existing `Program.cs` location in the build directory.
- The service writes the new `Program.cs` content (from the template) to the same path.
- The service returns 0 on success and a non-zero exit code on failure.
- The service logs informational messages about the modification.
